## [Unreleased]

### Sidebar — Active factory indicator

- The sidebar's factory list now shows which factory you're currently looking at: an orange bar (the same orange as the selected tab's underline) on the left edge of that factory's entry, following you as you scroll the plan or jump between factories — no more losing your place in a 30-factory plan. A factory counts as "in view" when its card spans an eye-line 10% down the planner pane (not the pane's very top edge), so a factory scrolled slightly past its header is still credited as the one being viewed rather than the tail end of the factory above it. The Statistics and Factories Summary jump-links get the same treatment when their sections are in view, and while the eye-line crosses the divider gap between two cards the indicator stays put on the last entry instead of blinking off. Works in both the desktop sidebar and the mobile navigation drawer.
- The thick divider between factory cards in the main view swapped its blue for the app header's burnt orange (now a shared `--sf-header` token, so the header, its gold border and the divider draw from one definition in `colors.ts`).
- In the Factories Summary sidebar row, the factory count chip and the fullscreen-summary expand control swapped places: the expand control now sits flush at the row's right edge, sized to the same 30px column as the factory rows' game-sync cells so the two line up down the sidebar.

### Export Calculator — Belts

- The Export Calculator gained a Belts transport method (returning from the pre-rework calculator, now per-destination): a full-width section beneath Train/Drone/Truck/Tractor showing how many conveyor belts of a chosen mark it takes to carry the export. All six marks (Mk.1–Mk.6 at 60/120/270/480/780/1,200 items per min) are selectable via an icon toggle, and the default mark auto-picks the smallest single belt that can carry the amount. Belt speeds are maintained in code — they aren't part of the game docs data. Fluid exports point to pipelines instead.
- Belt groups: an export can be split across multiple groups of belts, each with its own mark — group columns fill left to right (up to four per row, divided like the transport columns, each group's controls centred within its column). Each group offers two linked inputs — an items/min amount (with the shipped item's icon, ingredient-chip style) **OR** a whole number of belts to build (with the belt icon): entering an amount derives the belts to build (rounded up), entering a belt count sets the amount to those belts' full capacity, so players can plan by exact throughput or by "I'll just run N belts". Over-provisioning is fine; only undercapacity (groups carrying less than the export) raises the red warning chip — otherwise a green "All good" tick shows beside the actions. Between the two sits an amber nudge for structurally redundant groups: if a whole group could be deleted with the remaining groups' built belts still covering the export (judged on whole-belt capacity, e.g. 8+1+1+1 belts for a 10-belt export, or one group whose belts already carry everything), the chip says how many groups are redundant — deliberate headroom where every group is still needed is never flagged. Adding a group never rebalances fine-tuned amounts — the new group just picks up whatever the export still lacks (0 if fully allocated), with "Split evenly" available to rebalance on demand; per-mark total chips sum the whole belts to build across groups (three groups of 3× Mk.5 totals 9, not the ceiling of the fractional sum). A lone group tracks the export amount (never shrinking a deliberate over-provision on reopen). Groups persist with the plan (saved per part + destination factory on the export calculator settings, so they ride along in save/load/share; older saves initialize lazily).
- **Pipes for fluid exports**: fluid exports swap the Belts section for a Pipes section with the same group mechanics — Mk.1 (300 m³/min) and Mk.2 (600 m³/min) pipelines, pipe groups with even splitting, the All good / mismatch status and per-mark totals. Pipe groups persist separately from belt groups on the same export calculator settings. Pipeline Mk.1/Mk.2 icons pulled from the wiki into the building assets (64/256px).
- Fluid Trucks (added to the game in patch 1.2, 3,200 m³) are now supported: the Truck column calculates fluid exports as "Fluid Trucks" with the proper Fluid Truck (tanker) icon — new vehicle asset pulled from the wiki — instead of telling you to package the fluid. Drones and Tractors genuinely cannot carry raw fluids (drones/tractors have no fluid variant; packaged fluids are solids and travel as normal cargo), so for fluid exports those two columns are greyed out with a note saying to package first. The Fluid Freight Car stays at 1,600 m³.
- Tidied the Export Calculator tray: it now reads as an actual recessed tray (darker background with an inset lip, breathing room top and bottom), and the mixed thick/thin borders between the transport columns were unified into single thin dividers matching the satisfaction table's row lines — the dividing lines under the accuracy chips are gone, so each column's header visually attaches to its content. The tray also grows with its content instead of clipping at a fixed 250px height (scrolling beyond ~800px).
- Calculator polish: the Train's round-trip input icon is now a train instead of a refresh glyph; the Drone column gained a disabled "N/A" placeholder where the other vehicles have Start Timer (its rows now line up, helped by no longer reserving validation-message space under the drone time input); the belt group delete button matches the factory header's square outlined red trash button; and `tooltip-info` icons no longer sit ~3px below the surrounding text's baseline (FontAwesome's -0.125em shim, applied app-wide).

### Fixes

- The hidden sidebar's hover tray no longer sticks open when the cursor leaves the window — flinging the cursor onto a monitor to the left (or alt-tabbing to the game) used to trip the left-edge peek zone on the way out and leave the tray showing indefinitely. A peek triggered by the cursor exiting the window is now provisional (it closes after a second unless the cursor comes back in), and the tray always collapses when the window loses focus. Peeking by hovering the left edge inside the window behaves exactly as before.
- Copying a factory that supplies other factories no longer throws an error during the recalculation that follows: the copy carries the original's dependency requests, and cleaning up such an orphaned request could pull the entry out from under the cleanup scan itself mid-loop.

## Beta v0.5 — The "Overclocked" Update

After a long hiatus: the highly anticipated Overclocking and Somersloop support via Building Groups, a huge Power update, an item-centric Parts & Recipes browser, and a planner-wide UI refresh.

### Building Groups — Overclocking & Somersloops

- Each product and power generator can now be split into Building Groups, letting you plan how production lines are physically laid out in your world. Each group has its own building count and clock speed, and the planner keeps them in sync with the production targets.
- Implemented Somersloops (production amplification) on product building groups: per-building somersloop input clamped to each building's slot count (smelter/constructor 1, assembler/foundry/refinery/converter 2, manufacturer/blender/particle accelerator/quantum encoder 4; packager and generators cannot be amplified). Output (product + byproducts) is boosted by `1 + filled/slots`, power by `(1 + filled/slots)²`, both stacking with overclocking per the wiki formulas (fully slooped at 250% ≈ 13.43× base power). Ingredient consumption is never amplified — the item's ingredient demand is discounted to what the machines actually consume, and effective/remaining buildings, rebalancing, part reverse-solving and remainder actions are all somersloop-aware. Covered by a new unit suite (`building-groups/somersloops.spec.ts`) and TDD suite (`tdd/building-groups/somersloops.spec.ts`, refs `BG-E-S-PROD`).
- The "Open/Close Building Groups" bar shows the total Somersloops (products) and Power Shards (products + power producers) the item's groups consume, always visible with 0 when unused — both resources are rare, so their usage needs to be visible at a glance. Shards follow the game rule of 1 per building per 50% clock above 100% (150% = 1, 200% = 2, 250% = 3).
- Reworked how Building Groups are opened: the toggle moved from the cramped top-right control strip to a full-width "Open/Close Building Groups" bar underneath each product and power producer (chevron up when closed, down when open), which expands the groups as a tray below it (red with a warning when the groups have a problem). Inside the tray, the action buttons (Evenly balance, Remainder to last/new group, OC @ 100%) sit at the very top, followed by the status/sync row, then the groups, with "Add Building Group" centered at the bottom.
- Rewrote the Building Groups intro tooltip (shown on the closed toggle) to actually explain what Building Groups do, and shortened the Sync status tooltip (which also wrongly claimed sync re-enables when groups are reduced to 1 — it stays off).
- The "Open Building Groups" bar is disabled while the product has no item/recipe selected.
- Implemented `isEvenlyBalanced` logic for Product Groups to improve user feedback and action button states.
- Added comprehensive TDD tests for Building Groups: Creation, Deletion, Action Buttons, and Item Editing.

### The Power Update

- Added Geothermal Generators and Alien Power Augmenters as selectable power generators. Geothermal is fuel-less — you pick the geyser's node purity (Impure 100 MW / Normal 200 MW / Pure 400 MW average) and the planner shows the 0.5×–1.5× oscillation range; it cannot be overclocked and has no building groups (they'd only echo the building count). The Alien Power Augmenter produces 500 MW static per building and applies a grid-wide circuit boost — +10% of the plan's total generation per augmenter, or +30% when a building group is toggled to "Supply Matrixes", which also adds real Alien Power Matrix demand (5/min per augmenter) to the factory's satisfaction ledger so shortages surface like any other part. The boost (assuming one power grid, as most players run) is shown on the factory's Power & Buildings card and in the world power statistics, where generation now includes the boost to match the in-game power graph. Both buildings are emitted by the parser from the game docs (data regenerated as v1.2-03) rather than hardcoded.
- Reworked the world Power statistics from chips into a table that makes the circuit boost math explicit: base Generation, Circuit boost (own colour, with %), Total generation, Consumption (average, with the min–max swing range when variable-power buildings are present) and Difference. An advisory note recommends covering at least the average draw and absorbing spikes with Power Storage rather than generating for the peak. Alien Power Augmenter building groups are always kept in sync (there is nothing to fine-tune without clocks or fuel), and their tray hides the balancing actions and sync/effective-buildings status — only the groups themselves and "Add Building Group" remain.
- The world Power statistics table has Average / Minimum / Maximum columns: generation carries the variable-generator swing (Geothermal), the circuit boost scales with it, consumption shows its existing range, and the difference takes the averages with an asterisked footnote advising Power Storage coverage up to the maximum consumption.
- The power target is now plan-specific: it lives on the factory tab (travelling with save/load/share) instead of a single browser-wide value, with the old global value used as a fallback until the target is next changed.

### Parts & Recipes

- Revamped the Recipes page into an item-centric "Parts & Recipes" browser at `/parts` ([#224](https://github.com/satisfactory-factories/application/issues/224)); `/recipes` redirects there. Each part is an accordion showing the recipes that produce it (with amounts per minute, byproducts, and the building used with its power draw — previously recipes showed no products or building at all), alternate recipes tucked into a nested "Alternate Recipes" accordion, and a "Used in" accordion listing every recipe that consumes the part. Search is now fuzzy across both part and recipe names, with results split into Parts and Recipes sections, and every recipe card has an "Add to Planner" button that opens a dialog listing your factories — add the recipe to any of them (or a new factory) at the default rate of 1 building @ 100% clock, with a "Go to Planner" shortcut. The dialog highlights factories that already produce ("Produces this") or consume ("Uses this") the part. Parts produced by a factory in the current plan get an "In Plan - X/min" badge (totalling production across the plan), a "Produced in Plan" filter chip narrows the list to those parts, and the opened panel lists Exports-style factory chips (with per-factory /min) that jump straight to that factory in the Planner. Search is debounced and subsequence matching is constrained (word-boundary start + compactness) so e.g. "iron" no longer matches "Packaged Nitrogen Gas".
- Recipe cards now lay out in a responsive grid (three across on desktop, two on medium screens, one on mobile) instead of stacking full-width, and "Produced by" became its own accordion (open by default) alongside "Alternate Recipes" and "Used in". The accordions got a greyer background, light border and shadow so they stand out from the part panel, recipe cards inside are darker for contrast, and part entries in the list are now slightly lighter than the page background.

### Performance — the reactivity overhaul

- Massively reduced the reactive cost of recalculations. The calculation engine used to rebuild parts / metrics dictionaries directly on the live (reactive) plan on every pass — ~98% of those writes left values unchanged, yet each one still fired deep watchers (localStorage persistence, the sidebar factory list, and, catastrophically, Vue Devtools' sync store subscription, which deep-traverses the entire plan per write and made adding a factory hang for minutes on large plans). `calculateFactory` / `calculateFactories` now run the engine on a plain `structuredClone` of the plan and diff-commit only genuine changes back onto the live objects, preserving object identity throughout. A no-op recalculation now performs zero reactive writes (verified down from ~2,400 on the demo plan), a product edit fires ~30 watcher events instead of thousands, and `factoryUpdated` events are only emitted for factories whose data actually changed. Also fixed `setFactories` aliasing `previousInputs` to the live `inputs` array instead of snapshotting a copy.
- Added a browser-level reactivity test harness (`web/testing/browser/`) that drives the real planner (demo plan and Mael's MegaPlan) with Puppeteer, asserting DOM reactivity end-to-end and bounding the number of deep-watcher fires per interaction via a dev-only `window.__sfWatchCounter` hook.
- Added a 124-factory stress-test plan (4× Mael's MegaPlan with remapped ids, `factory-setups/stress-plan.ts`, loadable in dev via `window.__sfLoadStressPlan`) plus a browser stress harness (`stress.e2e.mjs`) that expands every factory, materializes ~294k DOM nodes, and measures edit→DOM-settle latency, DOM mutations, watcher fires and main-thread long tasks per edit. It confirms calculation churn is solved at scale (~37 fires, ~65 DOM mutations per edit; a no-op recalc of all 124 factories is 0 reactive writes / ~240ms) and pinpointed the next bottleneck: full-plan deep watchers re-traversing the whole plan per flush plus the persistence `JSON.stringify`, blocking edits ~4–6.5s on a 124-factory plan.
- Eliminated every full-plan deep watcher. Plan persistence is now event-driven instead of deep-watched: calculation commits (`factoryUpdated` / `calculationsCompleted`) trigger a debounced save that stringifies the raw tree (`toRaw` — several times faster than stringifying through proxies), explicit store mutations schedule saves directly, and direct mutations that bypass the calculator (renames, hidden toggles, tasks) are caught by a periodic compare-and-save plus a flush on `pagehide`/tab-hide — verified in the browser harness by reloading and asserting both a calculated edit and a non-calc factory rename survive. The sidebar factory list now watches an id fingerprint instead of deep-watching the plan, and the factories summary watches row count. Profiled result: Vue's deep-watch `traverse` no longer appears in edit profiles at all; edit→DOM-settle on the 124-factory stress plan dropped from ~5–6.5s to ~2.5–3.5s (measurement overhead included), with the remainder being component-update sweep and native style/layout across the ~294k-node DOM — the rendering rework's territory.
- Reworked input responsiveness around the faster engine. Numeric inputs now split "echo" from "recalculate": the typed value lands in the data instantly, and only the whole-plan recalculation is debounced — at 250ms (down from 750ms), with a consistent spinner beside every debounced input (product amounts, ingredients, byproducts, building counts, power producers, import quantities and building groups) so the pending recalc is visible. A/B-measured in-browser: raw no-debounce edits settle in ~90–180ms on the demo plan but ~580ms/keystroke on a 124-factory plan, and — decisively — without a debounce the recalc rewrites derived inputs (e.g. Requires quantities) mid-typing, corrupting entered values; the 250ms window is the guard against that (typing cadences ≤200ms/key are safe). A dev-only `window.__sfDebounceMs` override supports future A/B tests. Follow-up idea: suppress write-back to focused inputs, which would allow dropping the debounce entirely.
- Adding or removing a factory now persists and announces immediately (`factoryUpdated` + scheduled save) instead of waiting for the periodic safety net — previously a freshly added factory wasn't saved (or seen by sync) until up to 10 seconds later. The persistence safety-net interval is now 5s, and a dev-only `window.__sfLoadsCompleted` counter exposes the `loadingCompleted` event to browser tests.

### Fixes

- Quantities no longer come back a rounding-hair off what you typed (the reverse-solve rounding artifact). Typing a whole-number ingredient quantity (e.g. 1234 Crude Oil for Plastic) reverse-solves the product amount, which is often a repeating decimal (822.666…); capped at the game's 3-decimal precision, the recomputed ingredient then displayed as 1234.001 (or 1233.999 from older under-rounded data). Since exact round-trips are mathematically impossible with both sides quantised at 3dp, quantities within 0.002 of a whole number now snap to it (`snapNearInteger`) — at that precision it is vanishingly unlikely the user wanted 120.001 rather than 120. The snap is context-aware via a new `clockSetByUser` flag on building groups: when the user has dialled in a fractional overclock themselves (e.g. 223.333%), that precision is deliberate and derived quantities are kept exact — a converter at that clock still shows the game-accurate 535.999 Stone — whereas solver-derived fractional clocks (a typed quantity re-solving to 42 buildings @ 97.9365%) don't suppress the snap. It also never collapses tiny legitimate quantities (0.001 of a byproduct) to zero. Covered by unit tests on the snap math, engine round-trips and the user-clock exemption, plus a browser test typing into a real Requires input.
- Switching factory tabs no longer recalculates the whole plan. `initFactories` used to unconditionally re-run the calculation engine on every load, even though a tab's stored data is already fully calculated — on large plans this blanked the screen and hung for 4–5 seconds every time you changed tabs. Recalculation now only runs when a data migration actually backfilled a missing field (old saves); an already-current plan loads instantly. Also fixed a listener leak in `BuildingGroups.vue`, which registered a global `factoryUpdated` handler on mount but never removed it on unmount — leaked handlers accumulated across tab switches and multiplied the cost of every recalculation.
- Loading a plan template no longer zeroes out its fuel-based power generation. Producers defined only by a power amount (as plan templates are) got their default building group before any calculation had derived a building count, leaving a 0-building group — which the "building groups are sacrosanct" recalculation then faithfully synced the producer down to. A group set totalling zero buildings while its item has real ones is now treated as degenerate and rebuilt from the item, and the load-time migration no longer backfills groups for uncalculated producers. This also self-heals previously saved plans carrying such zero-groups.
- Fixed variable-power buildings (Particle Accelerator, Converter, Quantum Encoder) consuming almost no power in the planner (e.g. a Nuclear Pasta accelerator at 250% with full Somersloops showed 1.3 MW instead of ~13.4 GW). Two bugs compounded: the building-group power calculation read the flat buildings map, which only holds these buildings' 0.1 MW standby draw, instead of the recipe's real power; and the parser treated `mVariablePowerConsumptionFactor` as the maximum draw when it is actually the range *on top of* the constant (so Nuclear Pasta emitted 500–1000 MW avg 750 instead of the correct 500–1500 MW avg 1000). The parser formula is fixed (game data regenerated as v1.2-02), recipes now carry `minPower`/`maxPower`, and the planner calculates with the average while displaying the oscillation range — building group rows and product power chips show "avg (min – max)", and the factory Power & Buildings card plus the world power statistics show the peak draw alongside the average so grids can be sized against tripping. World power statistics now also sum the group-derived per-factory figures (accounting for overclocks and Somersloops) instead of the 100%-clock building requirements.
- Building groups are now sacrosanct during recalculation: the Recalculate action and plan loading never rebalance groups (counts, clocks, somersloops, matrix toggles) — when an item disagrees with its groups, the item's quantity is adjusted to match the groups instead, never the other way around. Regular item edits still rebalance groups as before (the sync feature), but an edit that doesn't change the effective building requirement now leaves a custom group split untouched. Guarded by a dedicated test suite (`building-groups/recalculate.spec.ts`).
- Editing a building group's output value no longer blows away a hand-set overclock. Previously, entering (or re-entering) an output on a group that was already overclocked would re-solve the whole group — e.g. 1 Converter @ 223.333% became 3 Converters @ 75% — because the solver preferred spreading the work across more buildings at a sub-100% clock. When an overclock is set, the group's building count is now preserved and only the clock is rescaled to hit the new output (falling back to a full re-solve only if that would exceed the game's 250% cap).
- Fixed a synced product's qty/min not matching its building groups to 3 decimal places. The item quantity was derived from the effective building count rounded to 4dp (e.g. 1 Converter @ 223.333% with a somersloop rounded to 3.35, giving 3.35 × 120 = 402), while the group produced 401.999; the quantity is now computed from the full-precision effective building count so it matches the group output exactly. A group's ingredient parts are likewise now computed straight from the recipe rate and the group's own clock rather than round-tripping through the item's amount-derived requirements, so they no longer drift by 0.001 when the item quantity changes. Covered by a Converter "solidification" spot-check (223.333% + one somersloop → 22.333 Reanimated SAM, 535.999 Limestone, 401.999 Iron Ore).
- Fixed the effective building count rounding each group to 3 decimal places when clocks support 4 (e.g. one Converter at 223.33% produced 267.96/min instead of the in-game-correct 267.996/min). Caught by a new Converter spot-check suite verified against in-game values, covering ingredients, Somersloop-amplified output and the variable power range at 250% with/without Somersloops and at 223.33%.
- Fixed editing a building group's building count updating the product's amount but not its ingredient requirements (or the factory's parts/power) until some other edit forced a recalculation. The calculation passes ran against the pre-writeback amounts and the gap was previously masked by a Vuetify labs `VNumberInput` quirk that spuriously triggered a second recalculation from the UI. `calculateFactory` now re-runs itself once when a building-group sync actually changes item amounts.
- Fixed the "Building Groups have a problem!" state falsely triggering when editing a group with sync ON (e.g. adding a Somersloop): the problem flag was computed before the group→item writeback and never refreshed, so a single calculation pass (what the UI runs) left it stuck red against stale item data. It is now recomputed after the writeback.
- Fixed group→item writeback (`checkForItemUpdate`) running on every recalculation, which replaced exact user-entered amounts with float-degraded values (e.g. 123 → 122.999…) and stomped the power producer's `updated` direction. It now only runs for edits originating from a building group.
- Fixed critical bug where `syncBuildingGroups` ignored the `buildingGroupItemSync` flag, causing unwanted rebalances.
- Fixed `deleteBuildingGroup` logic to maintain sync disabled after multiple groups are reduced to one, ensuring intentional imbalances are preserved.
- Fixed `BuildingGroups.vue` to ensure metrics are recalculated immediately on mount and on all relevant factory updates.
- Corrected default building count for newly added building groups to match current UI design (default 1).
- Updated "Evenly balance" button to work even when sync is disabled, providing better manual control.
- Fixed power producer building groups being excluded from the factory's total building requirements.

### Under the hood

- Migrated the repository to a pnpm workspace (per-package lockfiles) and moved to Node 24; Node 26 is also supported now that jsonwebtoken 9.0.3 dropped its transitive `SlowBuffer` dependency.
- Modernised the routing/plugin stack: vue-router 4 → 5 (the deprecated `unplugin-vue-router` was merged into vue-router core, so the separate package is gone and the Vite plugin now comes from `vue-router/vite`), the unmaintained `vite-plugin-vue-layouts` is replaced by its community successor `vite-plugin-vue-layouts-next`, and the unplugin trio jumped to their new versioning scheme (auto-import 21, vue-components 32, fonts 2 — ESM-only, config unchanged). Pinia 2 → 4 with `@pinia/testing` 2 (stores were already setup-style, so no code changes). Verified end-to-end in a real browser: routes, the `/recipes` → `/parts` redirect, layouts, and plan persistence across navigation all work with no console errors.
- Upgraded TypeScript 5.9 → 6.0 (the final JS-based line, semantically aligned with the new Go-native TS 7) and vue-tsc 2 → 3. TS 6 turns pre-TS-7 deprecations into errors, which drove some genuine cleanup: removed the deprecated `baseUrl` from tsconfigs (paths are anchored with `./` now; parsing's alias config was dead and simply deleted), moved the backend to `module`/`moduleResolution` `nodenext`, removed the long-warned-about `defineProps`/`withDefaults`/`defineEmits` compiler-macro imports from 11 components, and deleted two dead template refs vue-tsc 3 now correctly flags as unused. TS 7 itself waits until vue-tsc/typescript-eslint/ts-jest support its new API (expected after TS 7.1).
- Migrated linting to ESLint 9 flat config across all three packages. The dead `eslint-config-standard` / `eslint-plugin-node` / `eslint-config-vuetify` v1 stack is replaced by `neostandard` (standard's flat-config successor) plus `eslint-plugin-vue` 10 and `@vue/eslint-config-typescript` 14 in web, and `typescript-eslint` + `eslint-plugin-import-x` in backend/parsing. Rule behaviour is preserved (the v1 vuetify config's style rules were ported verbatim); the only code changes are auto-fixed spacing in TS type literals and two newly-caught nits (an unused catch binding and a ternary-as-statement).
- Upgraded the web build/test toolchain across two majors: Vite 5 → 7, Vitest 2 → 4 (plus @vitejs/plugin-vue 6, coverage-v8 4, jsdom 29, vite-plugin-vue-devtools 8, @types/node 24). This clears **all 37 known vulnerabilities** in the web package, including a critical arbitrary-file-read/execute hole in Vitest's UI server. Test-suite adaptations for Vitest's new mock semantics (`resetAllMocks` now restores real implementations and `spyOn` reuses existing mocks, so a few tests needed their spy history cleared explicitly).
- Upgraded every dependency across web, backend and parsing to its latest patch/minor version (highlights: Vuetify 3.7 → 3.12, Vue 3.5.40, Vue Flow 1.48, sass 1.101, Mongoose 8.24, Express 4.22, jsonwebtoken 9.0.3 — which fixes the backend crashing on Node 26, so Node 26 is now supported). Vuetify's `VNumberInput` graduated from labs along the way with two behaviour changes that needed adapting: it now rounds typed values to whole numbers by default (the planner configures unrestricted decimal precision globally, keeping fractional per-minute rates working), and it mis-handles the `v-model.number` modifier (silently dropping every keystroke), so the redundant `.number` modifiers were removed from all number inputs. (vue-router was held at 4.4.x during this pass, then upgraded to 5 in the routing-stack modernisation above.)
- `pnpm test` no longer runs Vitest in watch mode (it never finished in CI), and tests run with an in-memory `Storage` implementation under Node 24+.
