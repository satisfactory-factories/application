# Project overview

Satisfactory Factories is a logistics planner for the video game **Satisfactory**. The player splits their in-game production into "factories" (a megafactory floor, a microfactory, any meaningful chunk), declares what each factory produces and what it imports from other factories, and the tool builds a logistical graph across them. When demand changes anywhere (e.g. the player wants more of an end product), the effects trickle down through the graph and the tool highlights every factory that now has a bottleneck to resolve. Players can also mark a factory as "in sync" with what they have physically built in-game, and the tool flags when the plan drifts from that snapshot.

## The mental model in one paragraph

Everything revolves around an array of `Factory` objects held in a Pinia store and persisted to `localStorage` (optionally synced to a backend). Each factory maintains a per-part supply/demand ledger (`factory.parts`), recomputed by a deterministic calculation pipeline (`calculateFactory` / `calculateFactories`) every time the user edits anything. Imports on one factory become export demand on another (a `FactoryInput` on the consumer creates a `FactoryDependencyRequest` on the producer), so a single edit can invalidate ledgers across the whole plan — which is why the global recalculation runs all factories, twice. Anything with `amountRemaining < 0` is a shortage; shortages roll up into `factory.hasProblem`, which is what the UI (cards and the Vue Flow graph) paints red.

## Core domain concepts

- **Factory** — the unit of planning. Holds products, byproducts, power producers (generators), imports (`inputs`), the parts ledger, dependency records, power totals, sync state, tasks and notes. See [calculation-engine.md](./calculation-engine.md).
- **Product** — a recipe applied at some output rate (amount/min). Ingredient demand is derived from the recipe and lands in the ledger as `amountRequiredProduction`.
- **Import / dependency** — a consumer factory declares `{factoryId, outputPart, amount}`; the producer records the matching request. Producer-side shortfall on a requested part is a cross-factory bottleneck.
- **Power producers** — generators that consume fuel (creating part demand) and produce MW. They can be solved from any of four directions (buildings, fuel, power, ingredient) depending on which field the user last edited.
- **Sync state** — `inSync: true/false/null` per factory. `null` = never matched to the real game; `true` = plan matches what's built in-game; `false` = plan has drifted since the snapshot. The snapshot lives in `syncState`/`syncStatePower`.
- **Building groups** — a newer subsystem (branch `11-product-building-groups`): a product's total building count can be split into groups of physical buildings, each with its own overclock percent (and, later, somersloops). See [building-groups.md](./building-groups.md).
- **Game data** — all recipes/items/buildings/power numbers come from a versioned JSON (`web/public/gameData_v*.json`) generated by `parsing/`. The version is pinned in `web/src/config/config.ts` (`dataVersion`).

## Where to start reading code

1. `web/src/interfaces/planner/FactoryInterface.ts` — every important type.
2. `web/src/utils/factory-management/factory.ts` — `calculateFactory` shows the entire pipeline in order, with comments.
3. `web/src/components/planner/Planner.vue` — how the UI bridges into the engine (`provide('updateFactory')`).
4. `web/src/utils/factory-setups/` — prebuilt example plans (many reproduce specific GitHub issues); the fastest way to see realistic data.
